## 第 0 章 引擎身份

### 0.1 范围

"策略数学引擎"指策略层 `Step()` 函数及其内部计算所遵循的数学契约。它包含:

- 资产结构定义
- 市场状态感知层(设计空间)
- 信号合成框架
- 微观调仓引擎(Sigmoid 动态天平)
- 宏观建仓引擎(设计空间)
- DeadBTC 释放机制
- 可进化参数(Chromosome)契约
- `StrategyInput` / `StrategyOutput` 接口

### 0.2 与 GA 引擎的关系

| 维度 | GA 引擎(框架文档) | 策略数学引擎(本文档) |
|---|---|---|
| 关心什么 | 染色体如何进化、如何打分、如何排序 | 染色体在单次 `Step()` 调用中如何被解读、如何决策下单 |
| 时间尺度 | Epoch(若干代) | 一次 cron tick / 一根 bar |
| 输出 | `ChallengerResultPackage` | `StrategyOutput`(订单意图 + 状态) |
| 触发者 | 任务创建 API | Cron Tick / Adapter.Evaluate |

### 0.3 策略层四个"设计空间"

策略文档(后续每个具体策略各自一份)必须填充四个设计空间:

1. **设计空间一**:市场状态感知层(必须定义"安静态")—— 见第 2 章
2. **设计空间二**:宏观引擎语义 —— 见第 4 章
3. **设计空间三**:染色体字段清单与硬边界 —— 见第 7 章
4. **设计空间四**:DeadBTC 释放条件与节奏 —— 见第 6 章

本文档定义"设计空间"的接口形态,不定义任何具体策略的填充内容。

### 0.4 关键约束(铁律级)

- `Step()` 内部**禁止读取墙钟**;所有"当前时间"来自 `StrategyInput.NowMs`(铁律 2)
- `Step()` 在回测与实盘中是同一份代码,内部禁止 `if isBacktest` 分支(铁律 1)
- 摩擦扣除**不在** `Step()` 内;在回测适配器或 Agent 成交回报中完成

## 第 1 章 资产结构三态

(对应 §I-2.1)

策略层的资产组合分为三种形态:

| 形态 | 语义 | 是否参与微观 Sigmoid |
|---|---|---|
| **DeadBTC** | 长期沉睡仓位,宏观引擎建仓后进入此态 | ✗ |
| **FloatBTC** | 浮动仓位 | ✓ |
| **ColdSealedBTC** | 冷封存仓位(可选,原型阶段可不实现) | ✗ |

### 1.1 派生公式

```
TotalEquity        = (DeadBTC + FloatBTC + ColdSealedBTC) × Price + USDT
ReserveFloor       = max(MinReserveUSDT, micro_reserve_pct × TotalEquity)
SpendableUSDT      = max(0, USDT − ReserveFloor)
CurrentMicroWeight = (FloatBTC × Price) / max(MicroEquityBase, ε)
```

- `micro_reserve_pct`:默认 `0.25`,属于可进化染色体字段(详细范围在策略文档定义)
- `MicroEquityBase`:策略层定义(典型为 `TotalEquity × something` 或 `FloatBTC × Price + SpendableUSDT`,具体见策略文档)
- `ε`:数值稳定保护,典型取 `1e-12`,防止零除

### 1.2 三态间的合法迁移

| 迁移 | 触发 |
|---|---|
| `USDT → DeadBTC` | 宏观引擎成交 |
| `DeadBTC → FloatBTC` | `ReleaseIntent`(由 SaaS 账本翻账,不下发 Agent) |
| `FloatBTC → USDT` 或 `USDT → FloatBTC` | 微观 Sigmoid 调仓成交 |
| `FloatBTC → ColdSealedBTC` | (可选)策略文档定义条件 |

非法迁移:`DeadBTC` ⇏ `USDT`(死币不能直接卖出),除非经 `ReleaseIntent` 翻成 `FloatBTC` 后再走微观循环。

## 第 2 章 市场状态感知层

(对应 §I-2.2)

> **【策略设计空间一】** 具体的市场状态分类、特征、判断逻辑,由各策略文档定义。

### 2.1 引擎层的最低要求

1. **必须定义一个名为 "安静态" 的状态**,用于第 5 章微观引擎中的粉尘订单归零判定。
2. **状态判断在 `Step()` 内完成**,不读取墙钟,所有时间信息来自 `StrategyInput.NowMs`(铁律 2)。

### 2.2 输出契约

策略层在 `Step()` 内部计算出一个 `MarketState`(枚举或结构),用于:
- 第 3 章信号合成:可以驱动权重 `aᵢ` 的状态依赖切换
- 第 5 章微观引擎:控制粉尘订单是否归零(安静态归零)
- 第 5 章微观引擎:提供 `MarketBetaMultiplier`(状态对调仓激进度的调节)

### 2.3 接口建议(非强制)

策略实现者可在 `RuntimeState` 中记录 `MarketState` 历史用于诊断,但**`Step()` 不得依赖外部持久化的 `MarketState`** —— 每次 `Step()` 调用必须能基于 `StrategyInput` 重算出当前 `MarketState`(为了铁律 1 的回测同构性)。

## 第 3 章 信号与目标函数框架

(对应 §I-2.3)

### 3.1 Signal 标量

策略的"市场观点"通过一个**无量纲的** `Signal` 标量传递给微观引擎:

```
Signal = a₁·X₁ + a₂·X₂ + a₃·X₃ + ...
```

其中:
- `Xᵢ`:策略层定义的无量纲特征(归一化偏离度、动量、波动率比率等)
- `aᵢ`:合成权重,**属于染色体的一部分**,由 GA 在历史数据上搜索最优值

### 3.2 符号约定

| Signal 符号 | 语义 | 与 Sigmoid Exponent 的关系 |
|---|---|---|
| `Signal > 0` | 倾向减仓 | Exponent 增大 → TargetWeight < 0.5 |
| `Signal = 0` | 中性 | TargetWeight 由 InventoryBias 单独主导 |
| `Signal < 0` | 倾向加仓 | Exponent 减小 → TargetWeight > 0.5 |

**这一方向约定与第 5 章 Sigmoid 公式严格一致**。策略文档定义 `Xᵢ` 时必须遵守此符号约定;若策略的原始特征是反方向的(例如"动量大 → 看多"),则在合成时取负号。

### 3.3 无量纲化要求(铁律 5 推论)

`Xᵢ` 必须是**无量纲**的。常见构造:
- 价格偏离度:`(price - ema) / ema`
- 对数收益率:`log(price_t / price_{t-k})`
- 波动率比率:`σ_short / σ_long`
- Z-score:`(value - mean) / std`

**禁止**:把原始价格 / 原始成交量 直接当 `Xᵢ` 喂入 Signal —— 这会违反铁律 5,且让 `aᵢ` 失去跨标的可比性。

## 第 4 章 宏观引擎

(对应 §I-2.5)

> **【策略设计空间二】** 具体宏观建仓节奏由策略文档定义。

### 4.1 引擎层约定(契约边界)

1. **只买不卖原则**:宏观引擎原则上只产生买入意图;成交后仓位记入 `DeadBTC`,不进入 `FloatBTC` 微观调仓循环。
2. **死线兜底机制**:极端市况下(策略文档定义)必须有强制建仓回退,确保不会无限期空仓持币。
3. **不参与 Sigmoid**:宏观成交后的 `DeadBTC` 不被第 5 章的微观调仓触碰。

### 4.2 输出形式

宏观引擎在 `Step()` 内产生意图后,挂入 `StrategyOutput.MacroOrders`,字段为 `OrderIntent` 数组。`OrderIntent` 的具体字段见第 8 章。

### 4.3 与 DCA 双基准的关系(引擎层 / fitness 层)

GA 评估时,`fitness.AggregateScoreTotal` 会用一对 DCA 基准(详见框架文档 §6.6)作为 baseline 对照。策略层的宏观引擎**不感知**这两个 baseline 的存在 —— baseline 是评估框架内部的对照逻辑,策略层只负责按自己的逻辑下宏观订单。

## 第 5 章 微观引擎:Sigmoid 动态天平

(对应 §I-2.4 —— 微观引擎核心创新)

### 5.1 本质:仓位弹簧系统

Sigmoid 动态天平用 Sigmoid 函数实时计算**目标持仓权重**,通过买卖浮动仓使实际权重趋近目标权重。它是一个**与信号来源无关**的通用框架。

### 5.2 核心公式

```
CurrentWeight  = FloatBTC × Price / TotalEquity

EffectiveBeta  = max(0.01, β × MarketBetaMultiplier)
InventoryBias  = clamp(CurrentWeight, 0, 1) − 0.5
Exponent       = EffectiveBeta × Signal + γ × InventoryBias

TargetWeight   = 1 / (1 + e^Exponent),  clamp(0, 1)
```

### 5.3 公式解读

| 场景 | Signal | Exponent 方向 | TargetWeight | 动作 |
|---|---|---|---|---|
| 信号看空 | 正 | 增大 | < 0.5 | 减仓 |
| 信号看多 | 负 | 减小 | > 0.5 | 加仓 |
| 仓位 > 0.5,γ > 0 | 任意 | 额外增大 | 进一步压低 | 均值回归 |
| 仓位 < 0.5,γ > 0 | 任意 | 额外减小 | 进一步拉高 | 均值回归 |

### 5.4 参数语义

| 参数 | 语义 | 边界 |
|---|---|---|
| **β** (激进系数) | 越大调仓越频繁 | 染色体字段,具体范围见策略文档 |
| **γ** (仓位偏置系数) | `γ=0` 时纯信号驱动;`γ>0` 叠加均值回归力 | 染色体字段 |
| **MarketBetaMultiplier** | 来自市场状态感知层 | 由策略文档定义 |
| **EffectiveBeta 下限 0.01** | 防止 β 退化为 0 导致 TargetWeight 恒为 0.5 | 硬约束 |

### 5.5 理论订单与楔形过滤

```
DeltaWeight    = TargetWeight − CurrentWeight
TheoreticalUSD = DeltaWeight × TotalEquity
```

**粉尘拦截规则**(防止无效小额交易):

- `|TheoreticalUSD| ≥ 最小阈值`: 直接下单
- `|TheoreticalUSD| ∈ (0, 最小阈值)`: 仅在**非安静态** **且** 满足楔形突破条件时强制最小订单,否则归零
- 楔形突破条件: `|DeltaWeight| ≥ 阈值` **或** `VolatilityRatio ≥ 阈值`

### 5.6 VolatilityRatio 公式

```
VolatilityRatio = clip(MAV_short / MAV_long, 0.1, 3.0)
```

其中 `MAV` 是**平均绝对涨跌**(非 ATR):
```
MAV_window = mean(|Close[i] - Close[i-1]|, i in window)
```

> **说明**:`MAV` 与 `ATR` 的核心差别 —— `ATR` 用 high/low/close 的真实波动幅度,`MAV` 只看 close-to-close 的绝对涨跌。Sigmoid 微观引擎选用 `MAV` 是为了简化且足够稳健。

### 5.7 输出形式

微观引擎在 `Step()` 内产生意图后,挂入 `StrategyOutput.MicroOrders`,字段为 `OrderIntent` 数组。

### 5.8 摩擦的处理位置

**`Step()` 内不扣摩擦**。`TheoreticalUSD` 是不考虑摩擦的理论订单。摩擦(taker fee / slippage)扣除发生在:
- 回测:`adapters/backtest` 内的撮合模拟
- 实盘:Agent 成交回报

这一边界是铁律 1(策略同构)的具体表现 —— 回测/实盘里 `Step()` 完全相同,只在外部框架层处理摩擦。

## 第 6 章 DeadBTC 释放规则

(对应 §I-2.6)

> **【策略设计空间四】** 具体释放条件与节奏由策略文档定义。

### 6.1 引擎层约定

1. **释放意图通过 `StrategyOutput.ReleaseIntents` 返回**
2. **不向 Agent 下发为交易指令** —— Release 是账本翻账,不是市场交易
3. SaaS 侧收到 `ReleaseIntents` 后,直接修改内部 `PortfolioSnapshot` 的 `DeadBTC` 与 `FloatBTC` 字段(以等量翻转)

### 6.2 ReleaseIntent 字段(契约)

```go
type ReleaseIntent struct {
    Amount       float64  // 翻转的 BTC 数量
    Reason       string   // 触发条件标签(策略文档定义枚举)
    NowMs        int64    // 触发时间戳,来自 StrategyInput.NowMs
}
```

### 6.3 不变式

- Release 不通过交易所,不产生 fee 或 slippage
- 单次 Release 数量必须 ≤ 当前 `DeadBTC`
- Release 后 `DeadBTC + FloatBTC + ColdSealedBTC` 不变(账本守恒)

## 第 7 章 可进化参数契约(Chromosome 设计空间)

(对应 §I-2.7)

> **【策略设计空间三】** 具体字段清单由策略文档定义。

### 7.1 策略文档必须提供的清单

1. **字段清单**:名称 / 类型 / 语义 / 默认值 / 边界
2. **硬边界**:Clamp 修复规则
3. **结构约束**:块内约束、权重和归一化、互斥项等
4. **染色体 vs SpawnPoint 区分**:哪些进入基因组参与代内交叉变异,哪些是 Epoch 级冻结的出生点条件

### 7.2 染色体 vs SpawnPoint(关键区分)

| 维度 | Chromosome(染色体) | SpawnPoint(出生点) |
|---|---|---|
| 参与块级交叉 | ✓ | ✗ |
| 参与高斯变异 | ✓ | ✗ |
| 进入 Fingerprint 量化 | ✓ | ✗ |
| Epoch 内是否变化 | 每代演化 | Epoch 级冻结 |
| 典型示例 | β,γ,信号合成权重 `aᵢ`,`micro_reserve_pct` | `CapitalPolicy`,`RiskBounds`,初始资金 |
| 结果包归属 | `core.champion_gene` | `core.spawn_point` |

### 7.3 Segment 划分(与 GA 块级交叉强相关)

策略必须通过 `Segments()` 接口返回 `[]SegmentInfo`,定义"语义耦合基因块":

```go
type SegmentInfo struct {
    Name              string    // 唯一命名,例如 "macro_dca", "micro_dynamics"
    Dimensions        []int     // 该 Segment 内的基因维度索引
    QuantizationStep  []float64 // 各维度 Fingerprint 量化精度
    GeneStep          []float64 // 各维度 Mutate 步长
    IsCritical        bool      // 是否参与邻域稳定性 OAT 测试(三期)
    Description       string    // 文档说明
}
```

### 7.4 Segment 约束(框架文档 §2.2 同步)

- 所有基因维度必须被覆盖,任一维度只能出现在**一个** Segment 中
- `len(Dimensions) == len(QuantizationStep) == len(GeneStep)`,三者长度必须一致
- Segment 顺序必须稳定(同一实例生命周期内 `Segments()` 返回相同结果)
- 每个 Segment 含 **2–10 个维度**;单维度 Segment 等价于无块结构,应避免
- 必须覆盖测试:`TestSegmentsCoverage`、`TestCrossoverBlockFidelity`

### 7.5 Clamp 顺序

`Clamp` 调用顺序为:
1. 边界裁剪(单维度上下界)
2. 块内约束修复(例如权重归一化)
3. 跨段约束修复(例如互斥项)

详见框架文档 §3.3 块级交叉与 §5.1 接口表。

## 第 8 章 StrategyInput / StrategyOutput 契约

(对应 §I-2.8)

### 8.1 StrategyInput 字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `NowMs` | `int64` | 当前时间戳(毫秒),由调用方注入,`Step()` 内部**唯一时间源** |
| `Closes` | `[]float64` | 收盘价序列 |
| `Timestamps` | `[]int64` | 对应时间戳(毫秒) |
| `Portfolio` | `PortfolioSnapshot` | 资产三态快照(DeadBTC / FloatBTC / ColdSealedBTC / USDT) |
| `Params` | `Chromosome + SpawnPoint` | 当前冠军参数包 |
| `LastProcessedBarTime` | `int64` | 上次处理的 bar 时间戳 |

### 8.2 StrategyOutput 字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `MacroOrders` | `[]OrderIntent` | 宏观下单意图 |
| `MicroOrders` | `[]OrderIntent` | 微观下单意图 |
| `ReleaseIntents` | `[]ReleaseIntent` | DeadBTC → FloatBTC 转换意图,不下发 Agent |
| `RuntimeState` | 策略内部状态 | 持久化用 |
| `DebugSnapshot` | optional | 调试用,含 `Signal` / `TargetWeight` / `MarketState` 等 |

### 8.3 OrderIntent 字段(草案,与 Agent 端 TradeCommand 对齐) [部分 INVENTED v1]

```go
type OrderIntent struct {
    Kind            OrderKind   // Macro / Micro
    Side            OrderSide   // Buy / Sell
    OrderType       OrderType   // Market / Limit
    QuantityUSD     float64     // 计价金额(USD)
    LimitPrice      float64     // 可选(Limit 单使用)
    ClientOrderID   string      // 幂等键,Agent 用于去重
    ValidUntilMs    int64       // 过期不下单,典型值 NowMs + 60000
}
```

> **[INVENTED v1 — needs architect review]**:`OrderIntent` 在 Part I §I-2.8 仅以列名形式提及,字段细节为本文档草拟。Phase 4(策略模块)开工前需要冻结此结构。

### 8.4 RuntimeState 持久化

- `RuntimeState` 由策略自定义,SaaS 侧不解读其内部字段
- SaaS 在每个 cron tick 完成后将 `RuntimeState` 持久化到 DB
- 重启后从 DB 恢复 `RuntimeState` 注入下一次 `StrategyInput`
- **回测期**:每个 `Step()` 调用的 `RuntimeState` 是上一根 bar 输出的;Adapter 在 `Reset(plan)` 时清空 `RuntimeState`(铁律:`Reset` 不得携带与 Gene 相关的缓存)

### 8.5 DebugSnapshot 用途

- 调试与可视化(`Signal`、`TargetWeight`、`MarketState`、内部诊断字段)
- `Step()` 内对 `DebugSnapshot` 的填充必须**纯函数化**,不能依赖外部状态
- 在生产 SaaS 路径中,`DebugSnapshot` 默认不持久化(避免 DB 膨胀);仅在 `app_role=lab` 或显式诊断标志位下持久化
