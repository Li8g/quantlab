## 第 0 章 架构哲学

### 0.1 核心命题

QuantLab 是面向量化交易投资者的 **全天候智能量化管理 SaaS**。系统命题分三层:

1. **数学命题**:用遗传算法在历史 K 线上搜索一组在四个时间窗口(6m/2y/5y/10y)上具备"持续盈利能力 + 抗回撤"的染色体。
2. **工程命题**:把搜索过程、评估过程、人工 Promote 流程、实盘执行流程,封装成可观测、可重放、可治理的 SaaS。
3. **运营命题**:用户只看 Champion 的运行结果,不接触染色体细节;Challenger 是人类专家审查后才能晋升的候选。

### 0.2 两层硬边界

| 层 | 包 | 职责 | 禁止 |
|---|---|---|---|
| **引擎层** | `engine`, `fitness`, `verification`, `data`, `repository`, `api`, `resultpkg`, `quant` | 种群生命周期、窗口构建、调度、结果交付、Promote 流程 | 读取策略内部字段名、信号公式、仓位三态语义 |
| **策略层** | `strategy`, `strategies/<name>`, `adapters/backtest` | 基因语义、Clamp/Validate、`Step()`、`Evaluate` | import `engine` / `fitness` 内部细节;在 `Step()` 内读墙钟/做 I/O |

引擎只通过 `EvolvableStrategy` 接口(14 verb)操作 `Gene`。

### 0.3 三个核心承诺

1. **确定性**:同输入同输出(容差级,见框架文档 §7.4)
2. **可复现**:任一 Challenger 可在相同数据 + 版本下重放
3. **审计可达**:每个 Challenger 携带完整的 `reproducibility_metadata`(`epoch_seed`、`hardware_signature`、`go_version`、`build_id`)

### 0.4 不做的事(原型期)

- 实盘自动 Promote(始终人工)
- 多策略并发进化(单进程单策略)
- 跨标的相对价值(铁律 5)
- Web 前端(Part III)
- 完整 Prometheus 矩阵(精简版,Phase 12)

## 第 1 章 三端物理部署形态

QuantLab 由三个物理端构成。三端通过 Postgres + WebSocket + REST API 通信。

```
┌─────────────────────────┐    REST/HTTP    ┌───────────────────────┐
│       LAB (人类专家)     │ ──────────────▶ │      SAAS (云端)       │
│  - Web 浏览器 / curl     │                 │  - HTTP API (gin)      │
│  - 创建 EvolutionTask    │ ◀────────────── │  - GA Worker Pool      │
│  - 审查 Challenger       │   challenger    │  - Promote 决策入口    │
│  - Promote / Retire      │     report      │  - WebSocket Hub       │
└─────────────────────────┘                 │  - Cron Tick 驱动      │
                                            │  - Postgres + Redis    │
                                            └───────────────────────┘
                                                       ▲
                                            WebSocket  │  TradeCommand
                                            (双向)      │  DeltaReport
                                                       ▼
                                            ┌───────────────────────┐
                                            │   AGENT (本地)         │
                                            │  - 持有 API Key (铁律 3) │
                                            │  - 撮合下单            │
                                            │  - 心跳 / 状态上报     │
                                            │  - 1:1 服务一个账户    │
                                            └───────────────────────┘
```

### 1.1 SaaS 端(`cmd/saas`)

**部署形态**:Linux 容器,长期运行,单实例(原型期不做水平扩展)。

**职责**:
- HTTP API(`POST /api/v1/evolution/tasks` 等,见 `CLAUDE.md`)
- GA 调度:Worker Pool 执行 Epoch 循环
- WebSocket Hub:维护与各 Agent 的双向长连接
- Cron Tick:1m 节拍驱动 `Step()` 调用(实盘模式)
- 结果包持久化、Promote 流程
- 数据所有权:Postgres + Redis(见 §4)

**对外端口**:`8080`(HTTP) `8081`(WebSocket) `9090`(Prometheus metrics)

**进程内组件依赖图**:
```
HTTP/gin ──┐
            ├──▶ EvolutionTask service ──▶ Engine (worker pool) ──▶ Adapter (backtest)
WS Hub ────┤                                    │
            │                                    └──▶ Repository ──▶ Postgres
Cron ──────┘
```

### 1.2 Agent 端(`cmd/agent`)

**部署形态**:用户本地机器或私有 VPS,1 个 Agent 服务 1 个交易所账户。

**职责**:
- 维持与 SaaS 的 WebSocket 长连接
- 接收 `TradeCommand` 消息,转换为交易所 REST/WS 调用
- 持有 API Key(配置文件 `config.agent.yaml`,**永不上送 SaaS** —— 铁律 3)
- 上报 `DeltaReport`(账户净值、持仓变化、订单状态)
- 心跳维持

**对外端口**:不监听任何端口(只发起出站连接到 SaaS WS)。

**铁律 3 推论**:Agent 端必须独立部署。SaaS 进程无法访问交易所凭证 ⇒ SaaS 一旦被入侵也不会丢失资金。

### 1.3 Lab 模式(`cmd/saas` 同二进制,`APP_ROLE=lab`)

**部署形态**:开发者本地机器,与生产 SaaS 用**同一二进制**,通过 `APP_ROLE` 环境变量切换行为。

**职责**:
- 跑历史回测、调试策略、验证 GA 收敛
- **不**接收外部用户请求
- **不**连接真实 Agent(回测 Adapter 全程在进程内)

**与生产 SaaS 的差别**:见第 2 章 `app_role` 行为矩阵。

## 第 2 章 `app_role` 三态行为矩阵

`APP_ROLE` 环境变量是 SaaS 二进制的运行模式开关。三态:`saas` / `lab` / `dev`。

| 行为开关 | `saas` (生产) | `lab` (回测/调试) | `dev` (本地开发) |
|---|---|---|---|
| 是否监听公网 HTTP | ✓ | ✓(仅 localhost) | ✓(仅 localhost) |
| 是否启用 WS Hub | ✓ | ✗ | ✓ |
| 是否启动 Cron Tick | ✓ | ✗(纯回测无 tick) | 可选,默认 ✗ |
| 是否允许 Promote API | ✓ | ✓ | ✓ |
| 是否允许 `test_mode=true` 任务 Promote | ✗(铁律) | ✗(铁律) | ✗(铁律) |
| Schema 迁移路径(默认) | goose | AutoMigrate | AutoMigrate |
| `bars_hash` 校验 | 严格 | 严格 | 严格 |
| Prometheus metrics 暴露 | ✓ | ✓ | ✓ |
| 启动校验三件套版本号 | 严格(不一致 → fail-fast) | 严格 | 严格 |
| 日志级别默认 | `info` | `debug` | `debug` |
| WS Agent 心跳超时 [INVENTED v1] | 30s | n/a | 60s |
| 数据库连接池上限 [INVENTED v1] | 50 | 10 | 10 |

> **Schema 迁移后端**由 `database.migration_mode` 决定,是**独立于 `app_role`** 的开关:`goose`(版本化迁移,生产保真)/ `automigrate`(GORM AutoMigrate + raw DDL)/ 空(按 role 推导:saas→goose、lab/dev→automigrate)。回测 / paper-trading 集群可设 `migration_mode: goose` 跑 prod schema 而不继承 saas 的其它行为;`app_role=saas` 不得用 automigrate。决策:`docs/saas-schema-migration-draft.md`(Goose 取代早期草案的 Atlas)。

> **不变规则**(无论 role): 铁律 1–6、三条隐含铁律、`bars_hash` 排除元数据、`sort.SliceStable` 强制、Adapter 状态隔离。这些不是配置开关。

## 第 4 章 全局状态总线

### 4.1 单一 Postgres 原则(铁律 6)

**Postgres + TimescaleDB 扩展是唯一真值源**。Redis 仅缓存,不持有真值,丢失后从 Postgres 重建。

### 4.2 数据所有权表

| 数据类别 | 存储 | 备注 |
|---|---|---|
| 业务实体(用户、策略实例、订单等) | Postgres | 标准 GORM 模型 |
| `EvolutionTask` 任务记录 | Postgres `evolution_tasks` | 含请求原始意图(`requested_taker_fee_bps`/`slippage_bps`) |
| 历史 K 线 | Postgres `klines` (hypertable) | 7 天 chunk,30 天压缩 |
| K 线缺口 | Postgres `kline_gaps` | 显式记录,**禁止** datafeeder 内插值 |
| `ChallengerResultPackage` | Postgres `challenger_result_packages` (JSON blob) | 五层结构 |
| `Challenger` 元数据 | Postgres `challengers` | `decision_status` 三态镜像 |
| `Champion` 历史 | Postgres `champion_history` | Champion 退役在此表独立管理 |
| `SharpeBank`(DSR 累积) | Postgres `sharpe_bank` | 跨 Epoch 累积,含 `spawn_mode`、`fingerprint_distance_to_parent` |
| 冠军基因缓存 | Redis `champion:{strategyID}` | 热路径 |
| 会话 / JWT | Redis | TTL 管理 |
| Agent 在线状态 [INVENTED v1] | Redis `agent:{accountID}:status` | TTL 60s,Agent 心跳刷新 |
| Worker Pool 任务队列 [INVENTED v1] | Postgres `worker_tasks` 或进程内 channel | 原型期用进程内 channel,持久化二期 |

### 4.3 Redis 失效策略

- **缓存 miss = 回源 Postgres**(永远不阻塞业务流程)
- Redis 重启后冷启动 ≤ 3 分钟内回温
- 任何"Redis 是真值"的代码路径都是 bug(铁律 6 推论)

### 4.4 TimescaleDB 配置

```sql
CREATE EXTENSION IF NOT EXISTS timescaledb;

SELECT create_hypertable(
  'klines',
  'open_time',
  if_not_exists => TRUE,
  chunk_time_interval => 7 * 24 * 60 * 60 * 1000  -- 7 天一个 chunk(毫秒)
);
```

`klines` 表唯一索引 `(symbol, interval, open_time)`,保证 `import` 命令幂等。

## 第 5 章 WebSocket 通信协议

> **本章大部分为 [INVENTED v1 — needs architect review]**。WebSocket 协议在现有基线中未指定;此处提出 v1 草案以便编码工作推进。Agent 端实现进入 Phase 7 之前必须人工审查并冻结。

### 5.1 通信原则

- **双向异步**,基于消息类型分发
- **JSON 编码**,带 schema_version 字段
- **幂等性**:`msg_id` 唯一,Agent 收到重复 `TradeCommand` 必须返回上一次的 `Ack`,不重复下单
- **断线自愈**:重连后双方先做状态同步(`StateSync` 消息)再恢复业务流

### 5.2 消息类型总表 [INVENTED v1]

| 方向 | 类型 | 用途 |
|---|---|---|
| SaaS → Agent | `TradeCommand` | 下发交易意图(MacroOrder / MicroOrder) |
| SaaS → Agent | `Ping` | 心跳探测 |
| SaaS → Agent | `StateSyncRequest` | 重连后要求 Agent 上报当前持仓 |
| SaaS → Agent | `KillSwitch` | 紧急停止该 Agent 的所有交易 |
| Agent → SaaS | `Ack` | 对 `TradeCommand` 的确认(含 exchange order id) |
| Agent → SaaS | `DeltaReport` | 持仓 / 净值变动 |
| Agent → SaaS | `Pong` | 心跳响应 |
| Agent → SaaS | `StateSyncResponse` | 完整持仓快照 |
| Agent → SaaS | `OrderUpdate` | 订单状态变化(filled / cancelled / rejected) |
| Agent → SaaS | `Error` | 异常上报(包含交易所错误码) |

### 5.3 消息信封 [INVENTED v1]

```json
{
  "msg_id": "uuid-v4",
  "type": "TradeCommand",
  "schema_version": "v5.3.3",
  "timestamp_ms": 1700000000000,
  "account_id": "uuid",
  "payload": { /* 类型相关字段 */ }
}
```

### 5.4 TradeCommand 字段 [INVENTED v1]

```json
{
  "intent_kind": "macro" | "micro",
  "symbol": "BTCUSDT",
  "side": "buy" | "sell",
  "order_type": "market" | "limit",
  "quantity_decimal": "0.00123456",       // 字符串保精度
  "limit_price_decimal": "65000.00",      // 可选
  "client_order_id": "uuid",              // Agent 用于幂等去重
  "valid_until_ms": 1700000060000,        // 过期不下单
  "now_ms_at_saas": 1700000000000         // SaaS 端 NowMs,供审计追溯
}
```

`quantity_decimal` / `limit_price_decimal` 用字符串传输,Agent 内用 `shopspring/decimal` 解析(铁律推论:跨边界用 string,内部用 decimal)。

### 5.5 DeltaReport 字段 [INVENTED v1]

```json
{
  "account_id": "uuid",
  "reported_at_ms": 1700000000123,
  "positions": [
    {"symbol": "BTC", "free_decimal": "0.5", "locked_decimal": "0.0"},
    {"symbol": "USDT", "free_decimal": "1000.0", "locked_decimal": "0.0"}
  ],
  "since_last_report": {
    "fills": [/* 自上次 DeltaReport 以来的成交列表 */],
    "errors": [/* 自上次以来的错误列表 */]
  }
}
```

### 5.6 状态收敛与自愈 [INVENTED v1]

**问题**:Agent 重连后,SaaS 无法假设其内部状态与断线前一致(可能错过 `TradeCommand`,可能在断线期间手动平过仓)。

**机制**:
1. WS 连接建立后,SaaS 立即下发 `StateSyncRequest`
2. Agent 在 1.5s 内回 `StateSyncResponse`(完整持仓快照 + 自上次心跳以来的全部成交)
3. SaaS 用快照 ground-truth 覆盖内存中的 `PortfolioSnapshot`
4. **超时未响应**:Agent 标记 `degraded`,暂停下发 `TradeCommand`,触发告警
5. **快照与 SaaS 持仓不一致**:写 `discrepancy_event` 入库 + Slack 告警,人工介入

### 5.7 心跳协议 [INVENTED v1]

- SaaS 每 10s 发 `Ping`
- Agent 必须在 5s 内回 `Pong`
- 连续 3 个 `Ping` 无响应 → SaaS 主动断开重连,Agent 进入"已知离线"状态
- Agent 持续重连尝试,使用指数退避(见第 6 章)

## 第 6 章 系统级生命周期动作

### 6.1 初始化序列(SaaS 启动)

```
1. 加载配置(internal/saas/config)
2. 启动 logger(zap)
3. 连接 Postgres → 按 `migration_mode` 走 goose 版本化迁移,或 AutoMigrate + raw DDL(含 TimescaleDB 扩展/hypertable)。空则按 app_role 推导:saas→goose、lab/dev→AutoMigrate(见 §2)
4. 连接 Redis
5. 启动校验:
   a. SchemaVersion        == "v5.3.3"
   b. FitnessVersion       == "v1-raw-std"
   c. FingerprintVersion   == "fp-v1"
   d. grep -rn '"v5.3.3"' internal/ | grep -v internal/resultpkg/versions.go  → 应无结果
   不一致 → fail-fast,进程退出
6. 初始化 Worker Pool(根据 APP_ROLE 与 NumCPU)
7. 启动 WS Hub(saas / dev)或跳过(lab)
8. 启动 Cron Tick(saas)或跳过(lab / dev)
9. 启动 HTTP server(gin)
10. 注册 Prometheus metrics endpoint
11. 注册 signal handler(SIGTERM / SIGINT)→ 触发 6.4 优雅停机
```

### 6.2 Cron Tick 驱动 Step() 完整流程(实盘模式) [部分 INVENTED v1]

> **铁律 2 关键边界**:`time.Now()` **只在 Cron Tick 外圈被调用一次**,得到 `NowMs`,然后注入 `StrategyInput.NowMs`,贯穿整个 `Step()` 调用链。`Step()` 内任何对 `time.*` 的调用都是 bug。

```
每分钟 cron 触发(默认 1m 节拍):
1. tick_now_ms := time.Now().UnixMilli()    // ← 唯一墙钟读取点
2. 对每个 active 策略实例:
   a. 从 DB 拉取最新 PortfolioSnapshot
   b. 从 DB 拉取最近 N 根 1m K 线(N 由策略 MinEvalBars 决定)
   c. 拼装 StrategyInput{NowMs: tick_now_ms, Closes, Timestamps, Portfolio, Params, LastProcessedBarTime}
   d. 调用 strategy.Step(ctx, input) → StrategyOutput
   e. 遍历 output.MacroOrders + output.MicroOrders:
      - 转换为 TradeCommand
      - 设置 valid_until_ms = tick_now_ms + 60000(下个 tick 失效)
      - 通过 WS Hub 推送到对应 Agent
   f. 遍历 output.ReleaseIntents:
      - 不下发 Agent
      - 直接调用 SaaS 内部账本翻账逻辑(DeadBTC → FloatBTC)
   g. 持久化 RuntimeState 到 DB
3. tick 完成,等待下一个 cron 触发
```

### 6.3 Agent 断线重连指数退避 [INVENTED v1]

Agent 侧维护重连状态机:

| 状态 | 触发 | 动作 |
|---|---|---|
| `connected` | 初始 / 成功握手 | 正常业务流 |
| `disconnected` | 检测到连接断开 | 标记时间戳 `disconnected_at` |
| `reconnecting` | 进入指数退避循环 | 见下表 |
| `fatal` | 重连失败超过 1 小时 | 告警 + 持续重试(不放弃) |

退避序列(毫秒):`500, 1000, 2000, 4000, 8000, 16000, 32000, 60000`(封顶 60s,然后保持 60s 间隔重试)。

每次重连成功后,执行 §5.6 的状态同步序列。

### 6.4 优雅停机与状态快照

收到 `SIGTERM` 或 `SIGINT`:

```
1. HTTP server 停止接受新请求(继续处理已建立连接,默认 30s 上限)
2. Cron Tick scheduler.Stop()
3. Worker Pool 通过 ctx cancel 中止当前 Evaluate;
   未完成的 Epoch 标记为 interrupted,保留中间状态供下次重启时检查
4. WS Hub 向所有 Agent 广播 graceful_shutdown 通知,Agent 进入 reconnecting 状态
5. 落盘当前 PortfolioSnapshot 与 RuntimeState
6. 关闭 Redis / Postgres 连接
7. flush 日志
8. 进程退出
```

**关键**:Worker Pool 中断后,部分完成的 Epoch 不写 Challenger,但已落盘的 PortfolioSnapshot / RuntimeState 必须可被下次启动直接续接。

## 第 7 章 不可推翻的技术决策

### 7.1 六条铁律

| # | 铁律 | 修订理由 |
|---|---|---|
| 1 | **策略同构**:回测与实盘共用同一 `Step()`,禁止 `if isBacktest` 分支 | 过去版本因分支差异导致回测验证不到实盘 bug;统一调用路径才能让回测真正代理实盘 |
| 2 | **策略纯函数**:`Step()` 禁读墙钟 / 禁网络 / 禁 DB / 禁文件 I/O。当前时间由 `input.NowMs` 注入 | 墙钟读取使回测不可复现;一旦混入 I/O,确定性测试就成空话 |
| 3 | **API Key 物理隔离**:凭证只存 `config.agent.yaml`,永不进 SaaS | SaaS 被入侵 ≠ 资金损失;符合零信任 |
| 4 | **Schema 双轨制**:开发期 `GORM AutoMigrate`,生产期切 **goose 版本化迁移**(后端由 `migration_mode` 选,独立于 `app_role`) | `AutoMigrate` 不能删字段/改类型、无只读模式,生产不可控;goose 提供版本化/review/回滚,drift 测试钉死两源真相 |
| 5 | **无量纲计算**:价格用对数收益率或比率,禁止跨标的绝对价格比较 | 跨标的绝对价比较没有物理意义;归一化是数值稳定性前提 |
| 6 | **单一 Postgres + TimescaleDB 扩展**:不分库;Redis 仅缓存;`klines` 表用 hypertable + 列存压缩 | 时序数据规模(千万行)需要列存;但分库会让事务管理塌方,扩展是折中 |

### 7.2 三条隐含铁律(v3 新增,同等强度)

| # | 铁律 | 关联测试 / grep |
|---|---|---|
| 7 | **CompareFitness 强制**:禁直接解引用 `*float64 SliceScore.Value` 排序;禁哨兵数值 | `TestCompareFitnessNilSafe` + `TestCompareFitnessFingerprintCollision` |
| 8 | **Adapter 状态隔离**:每个 worker 独占 Adapter 实例,`Reset(plan)` 由引擎每次 `Evaluate` 前**强制调用**,策略层无法绕过 | `TestAdapterResetIsolation` |
| 9 | **RNG 路径单一**:全引擎 RNG 从 `EpochSeed` 派生;禁 `rand.Seed()` 全局调用,禁直接用全局 `rand.New` | `TestEvaluateDeterministic` |

### 7.3 强制工程约束(语法级)

- 所有排序 → `sort.SliceStable`(`sort.Slice` 是 lint 禁项)
- 评估窗口循环 → 固定 `6m → 2y → 5y → 10y`
- 版本号读取 → `internal/resultpkg/versions.go`,禁字面量
- 浮点 → 回测层 `float64`,Agent 侧金额 `decimal.Decimal`
- 错误包装 → `fmt.Errorf("...: %w", err)`,禁裸 `panic` 跨层逃逸
- `bars_hash` 序列化 → `OHLCV + OpenTime`,**排除** `IsGap`/`GapType`(`TestBarsHashExcludesMetadata`)

## 第 8 章 时间语义(铁律 2 详解)

### 8.1 时间的唯一来源

```
NowMs 流动图:

回测模式:
  EvaluablePlan.Bars[i].OpenTime + IntervalMs  ──▶  StrategyInput.NowMs  ──▶  Step()

实盘模式:
  time.Now().UnixMilli() at cron tick outer ──▶  StrategyInput.NowMs  ──▶  Step()
            ↑
        唯一允许调用 time.* 的地点
```

### 8.2 禁止清单

`Step()` / `Evaluate` / `Adapter.Evaluate` 内禁止出现:
- `time.Now()` / `time.Since()` / `time.Now().Unix*()`
- `time.AfterFunc` / `time.Tick` / `time.Ticker`(回测期没有真实时钟流逝)
- 任何来自 `runtime.nanotime` 的间接调用

### 8.3 例外清单(允许调用墙钟的地点)

1. `cmd/saas/main.go` 启动序列(读启动时间作日志元数据)
2. `internal/saas/cron/*.go` Tick 外圈(读取 `NowMs` 并注入 `StrategyInput`)
3. `internal/api/*.go` HTTP handler(打 access log 时间戳)
4. `internal/repository/*.go` 写 `created_at` / `updated_at` 字段
5. `internal/saas/ws/*.go` WS 心跳与重连退避计时

**这些位置之外的 `time.*` 调用都是 bug**。CI 应配置 grep 检查:
```bash
grep -rn 'time\.Now\|time\.Since\|time\.Tick' internal/engine/ internal/strategy/ internal/strategies/ internal/fitness/ internal/verification/ internal/adapters/
# 期望: 无输出
```

### 8.4 回测与实盘的时间语义对齐

| 维度 | 回测 | 实盘 |
|---|---|---|
| `NowMs` 来源 | `Bar.OpenTime + IntervalMs` | `time.Now().UnixMilli()` at cron tick |
| `NowMs` 单调性 | 严格单调(按 Bar 顺序) | 单调但可能跳跃(网络抖动) |
| `NowMs` 间隔 | 严格 `IntervalMs`(默认 60000) | 大约 60000,允许 ±N ms 抖动 |
| `Step()` 看到的"未来 bar" | 严禁(future leakage) | 不存在 |
| 一次 `Step()` 调用对应的时间宽度 | 1 个 bar | 1 个 cron tick |

铁律 1(策略同构)在时间维度的具体含义:**`Step()` 内对时间的所有依赖,只能透过 `input.NowMs` 表达**。如果策略需要"过去 30 分钟",它从 `input.Timestamps` 反向找;如果策略需要"延迟下单到下一根 bar 开盘",它通过 `OrderIntent.ValidUntilMs` 表达,不能用 `time.AfterFunc`。
