## 0. 文档范围与设计原则

### 0.1 文档用途

本文档是进化系统的**程序框架总纲**，服务于以下活动：

- 统一开发团队对系统边界与模块职责的理解。
- 作为后续代码包结构、接口设计、数据库落表与 API 契约的依据。
- 为后续策略接入提供稳定的引擎层抽象。
- 作为测试计划、回放验证、版本迁移与审计扩展的基础。

### 0.2 系统边界

进化系统分为两层：

- **引擎层（Engine Layer）**：负责种群生命周期、数据窗口构建、适应度评估调度、结果交付、回放与 Promote 流程。
- **策略层（Strategy Layer）**：负责定义基因语义、约束修复、策略回测行为、具体适应度输入特征与验证逻辑。

引擎层必须对策略内部字段名、信号公式、仓位三态语义不可见。引擎只通过抽象接口操作 Gene。

### 0.3 设计原则

1. **黑盒解耦**：引擎不 import 具体策略实现细节。  
2. **确定性优先**：相同输入必须得到相同输出（容差级，见 §7.4）。  
3. **可复现性优先**：任一 challenger 必须能在相同数据和版本下重放。  
4. **服务化演进**：框架需支持任务提交、异步执行、结果查询、人工 Promote。  
5. **渐进扩展**：原型阶段先保证最小闭环；审计、长期 review、重度压力测试可后续扩展。  
6. **策略外置**：策略染色体表、策略公式、控制律参数均不写入引擎文档。  
7. **编码优先**：本文尽量采用可直接映射为模块、结构体、接口和 API 的表述。

## 1. 系统分层与模块划分

### 1.1 模块总览

建议将系统划分为以下模块：

- `domain`：核心领域对象与类型定义。
- `engine`：Epoch 执行、种群生命周期、调度主流程。
- `strategy`：EvolvableStrategy 抽象及具体策略适配实现。
- `fitness`：适应度计算与窗口聚合逻辑。
- `verification`：OOS、ReviewBacktest、MC、stress、DSR 等验证流程。
- `data`：K 线读取、Gap 处理、EvaluablePlan 构建。
- `repository`：数据库存取与结果包持久化。
- `report`：challenger 报告、验证报告、诊断输出。
- `api`：任务创建、状态查询、结果查询、Promote 接口。
- `tests`：单元测试、回放测试、契约测试、回归测试。

### 1.2 原型阶段推荐分层（Go-only server prototype）

原理验证版的服务器端建议**全部使用 Go 实现**，以降低跨语言接口与运行时差异带来的复杂度，优先保证确定性、可复现性、状态隔离和最小闭环落地。

职责划分如下：

- **Go（服务器主实现）**：
  - 主服务骨架与 HTTP API
  - 任务管理与 worker pool
  - Epoch 调度主流程
  - EvaluablePlan 构建
  - EvolvableStrategy 抽象接口与策略适配实现
  - Adapter 生命周期管理（`NewAdapter(plan)` / `Reset(plan)` / `Close()`）
  - Evaluate 调度与结果收集
  - fitness / OOS / ReviewBacktest 的服务端执行框架
  - ChallengerResultPackage 生成、结果落库与查询
  - 回放测试、契约测试、回归测试

- **Python（非服务器端，可选辅助工具）**：
  - 离线分析脚本
  - 可视化与实验后统计
  - 参数探索与报告草稿生成
  - 不进入服务器主链路，不承担在线任务调度、核心评估、结果持久化和 API 服务职责

采用 Go-only server prototype 的理由：

1. **减少跨语言 schema 对齐成本**：内部模块优先通过 Go struct 与 interface 协作，仅对 HTTP、数据库 JSON blob、结果包等外部边界维持稳定契约。  
2. **更容易保证确定性**：避免不同编程语言在浮点、空值、默认值、随机数边界上的差异干扰 `Evaluate`、Replay 和容差级一致性。  
3. **更容易实现状态隔离**：`Adapter.Reset()`、worker 独占实例、顺序无关性等约束可在单语言运行时内统一实现与测试。  
4. **测试体系更集中**：`TestEvaluateDeterministic`、`TestEvaluateOrderInvariance`、`TestAdapterResetIsolation`、`TestCascadeShortCircuit` 等关键测试可统一纳入 Go 测试体系。  
5. **符合当前阶段目标**：原理验证版优先完成“可运行、可重放、可诊断、可 Promote”的服务闭环，而非追求研究语言的灵活混合部署。

因此，本阶段文档中的 JSON schema 主要用于**API 契约、结果包结构和数据库持久化边界**，不再以 Python / Go 跨语言共享模型为主要目标。

### 1.3 原型阶段不建议过早实现的模块

以下能力建议放入二期或附录：

- 完整长期审计 API。
- Promote 后实盘追踪与月度再评估。
- 全量种群快照永久存储。
- 重型多服务拆分。

## 2. 核心领域对象

### 2.1 Gene / Chromosome

Gene 是固定长度的抽象向量，由策略层定义语义。引擎层不得直接读取内部字段名。

### 2.2 Segment

Segment 表示"语义耦合基因块"，用于块级交叉、Fingerprint 量化与变异步长配置。  
策略层必须通过 `Segments()` 接口返回完整的 `[]SegmentInfo`，而非简单的二维索引数组。

> **修订说明（审阅问题 B）：** 原草稿使用 `CrossoverSegments() [][]int`，仅能传递维度边界，会丢失 Fingerprint 量化精度（`QuantizationStep`）、Mutate 步长（`GeneStep`）和邻域稳定性标记（`IsCritical`）。改回 `Segments() []SegmentInfo` 以保留这三个已在 v5.2 落地的能力。

`SegmentInfo` 结构体定义：

```go
type SegmentInfo struct {
    Name              string    // 唯一命名，例如 "macro_dca", "micro_dynamics"
    Dimensions        []int     // 该 Segment 内的基因维度索引
    QuantizationStep  []float64 // 各维度 Fingerprint 量化精度
    GeneStep          []float64 // 各维度 Mutate 步长
    IsCritical        bool      // 是否参与邻域稳定性 OAT 测试（§6.9）
    Description       string    // 文档说明，用于诊断输出
}
```

约束要求：

- 所有基因维度必须被覆盖，任一维度只能出现在一个 Segment 中。
- `len(Dimensions) == len(QuantizationStep) == len(GeneStep)`，三者长度必须一致。
- Segment 顺序必须稳定（`Segments()` 在实例生命周期内返回相同结果）。
- 每个 Segment 含 2–10 个维度；单维度 Segment 等价于无块结构，应避免。

### 2.3 SpawnPoint

SpawnPoint 是 Epoch 级固定外生变量，包含：

- `CapitalPolicy`
- `RiskBounds`
- 其他策略运行所需、但不进入基因编码的出生点条件

SpawnPoint 不参与交叉、不参与变异、不进入 Fingerprint。

### 2.4 EvaluablePlan

EvaluablePlan 是单次 Epoch 内只读的评估上下文，至少包含：

- `Pair`
- `Spawn`
- `LotStep`
- `LotMin`
- `Windows`
- `DCABaselines`
- `OosWindow`（Anchored Holdout Window）
- `Friction`
- `AggregateCache`

原型阶段如不启用 Sacred Holdout，可先不加入 `SacredOosWindow`。

### 2.5 CrucibleWindow

CrucibleWindow 表示单个评分窗口，至少包含：

- 时间区间
- warmup 前缀
- 评估区间 bar 序列
- GapRanges（如启用缺口语义）

### 2.6 SliceScore / ScoreTotal

使用显式结构而非单个 float：

```go
type SliceScore struct {
    Fatal  bool
    Value  *float64 // Fatal=true 或被短路跳过时为 nil，禁止在此处写入哨兵数值
    Reason *string  // 可选，例如 mdd_exceeded / cascaded_skip
}
```

**SliceScore 的三态语义（CrucibleResult 上下文）：**

| 状态 | Fatal | SkippedBy | Value |
|---|---|---|---|
| 正常评估完成 | false | nil | 非 nil |
| 被级联短路跳过 | false | 非 nil | nil（合法）|
| 自身触发 Fatal | true | nil | nil（合法）|

三种状态互斥。`Fatal=false && SkippedBy==nil` 时 Value 必须非 nil；其他情况 Value 为 nil 是合法的。

**Fatal 排序约束（Go 实现层，v5.3.3 新增）：** `Fatal=true` 时 `Value` 为 nil，在 Go 的 `sort.Slice` 中直接解引用 nil `*float64` 会引发 Panic。引擎**必须**封装专用比较函数，三层优先级：

```go
// 禁止直接用 *a.Value > *b.Value 做排序
func CompareFitness(a, b ScoreTotal) int {
    if a.Fatal && b.Fatal { /* 平局：用 Fingerprint 字典序稳定排序 */ }
    if a.Fatal  { return 1  } // a 劣于 b
    if b.Fatal  { return -1 } // b 劣于 a
    // 两者均 Normal：按 Value 降序
}
```

**禁止**向 `Value` 写入 `-99999`、`-1e18` 等哨兵数值来规避 nil 判断，这会重新引入魔数依赖。对应强制测试：`TestCompareFitnessNilSafe`（见 §10.1）。

### 2.7 ChallengerResultPackage

原型阶段建议最少包含以下层：

- `core`
- `evaluation`
- `verification`
- `diagnostics`
- `promote`

`review` 可在二期再落地。

### 2.8 Gap-aware Bar

如采用缺失数据契约，则 Bar 结构建议扩展为：

```go
type Bar struct {
    OpenTime int64
    Open, High, Low, Close, Volume float64
    IsGap   bool
    GapType string
}
```

## 3. 运行时生命周期与状态机

### 3.1 Epoch 生命周期总览

建议将单次进化任务的生命周期定义为：

1. 创建任务  
2. 冻结 SpawnPoint  
3. 构建 EvaluablePlan  
4. 初始化种群  
5. 进入代内循环  
6. 输出 challenger  
7. 执行 OOS / ReviewBacktest（视配置）  
8. 等待人工 Promote  
9. champion 替换或保留现任 champion

### 3.2 创建任务

任务创建时需要固化：

- `pop_size`
- `max_generations`
- `elite_ratio`
- `fatal_mdd`
- `taker_fee_bps`
- `slippage_bps`
- `spawn_mode`（枚举：`inherit` / `random_once` / `manual`）
- `test_mode`（boolean：`true` = 烟雾测试模式）
- `oos_days`（可选整数：Anchored Holdout 天数，nil = 不启用）
- 当前引擎版本与数据版本快照

### 3.3 冻结 SpawnPoint

同一 Epoch 内，全种群必须共享同一 SpawnPoint，保证横向公平性。  
不同 Epoch 之间若 SpawnPoint 不同，`ScoreTotal` 不应跨上下文直接绝对对比。

### 3.4 初始化种群

建议默认值延续 v5.2 主干：

- 10% 精英克隆
- 40% 精英强化变异
- 50% 随机生成

如无历史精英：

- 索引 0 为默认冠军染色体
- 其余为随机生成

### 3.5 代内循环

每代执行顺序：

1. Evaluate population（并发，每个 worker 独占 Adapter）
2. Sort / Rank（主线程串行，基于 SliceScore Sum-type 语义）
3. Elitism 保留  
4. Tournament 选择  
5. Block Crossover  
6. Mutation  
7. Clamp / Validate  
8. 生成下一代

### 3.6 收敛检测与救援

建议保留"斜坡 + 重启动"的双层策略：

**层一：变异斜坡（应对适应度停滞）**  
触发条件：连续 N 代无改进（条件 A）。  
动作：提升 `MutationProbability` 与 `MutationScale`（每次 ×1.25），上限分别为 0.55 和 3.0。

**层二：多样性救援（应对种群塌缩）**  
触发条件：条件 A **且**种群基因熵低于历史滑动均值 50%（条件 B）。  
动作（参照 v5.2 §2.7.3）：

- 从 Top 30 个体中，用**贪心 max-min diversity 算法**选出 5 个相互 Fingerprint 距离最大化的核心精英保留。
- 剩余约 98% 按以下比例重新填充：60% 完全随机（`Sample(rng)`）、25% 对核心精英施加 5×GeneStep 强变异、15% 精英杂交后代。
- 救援后冻结 3 代不再触发收敛检测。

> **关于 98% 与"彻底重置"：** 这里"98%"指的是被替换的个体比例（约 295/300），但保留的那 2%（Top 5）并非随机选取，而是按最大 Fingerprint 距离主动筛选，确保保留的精英尽量不相似，避免"5 个克隆精英"退化。

原型阶段如觉得双层机制实现量太大，可先只实现层一，层二留接口预留。

### 3.7 交付 challenger

Epoch 结束后：

- 挑选最优 Gene（SliceScore.Fatal=false 且 ScoreTotal.Value 最高）
- 调用 `EncodeResult`
- 生成 ChallengerResultPackage
- 落库为 `challenger`
- 不自动 Promote

## 4. 数据与窗口构建

### 4.1 K 线读取

引擎启动 Epoch 时，应一次性读取本次所需的完整历史序列，而不是在代内反复查询数据库。

### 4.2 多窗口坩埚

建议保留四窗口结构：

| Window | Span | Weight |
|---|---:|---:|
| 10y | 全量最长序列 | 0.40 |
| 5y | 1825 天 | 0.30 |
| 2y | 730 天 | 0.20 |
| 6m | 183 天 | 0.10 |

实现约束：

- 权重为系统默认值（0.40/0.30/0.20/0.10 是 √T 加权的近似方便值，所有实现必须使用此预设值）。
- 评估区间前可附 warmup 前缀。
- 严禁 future leakage。

### 4.3 Warmup 规则

各窗口评估区间前可预留最多 1200 天 warmup bar，warmup 数据不得进入评分区间。

### 4.4 MinEvalBars

策略层可通过 `MinEvalBars()` 声明最低有效评估 bar 数。  
当某窗口不足时：

- 该窗口跳过
- 剩余窗口权重重新归一化
- 所有窗口均不足时直接返回 `SliceScore{Fatal: true}`

### 4.5 缺失数据契约

v5.3 建议采用"Gap-aware"语义，而非简单前值填充。原型阶段可采用最小实现：

- 允许记录 GapRanges
- Evaluate 遇 `IsGap=true` 的 bar 时跳过指标更新与订单执行
- GAP 期间不得撮合虚假成交
- 报告记录 gap 统计

若当前实现尚未支持 Gap-aware Bar，可在原型阶段暂用数据预处理层修补，但必须在文档中注明"临时兼容方案"。

### 4.6 缺口诊断

建议输出：

- `total_gap_minutes`
- `longest_gap_minutes`
- `gap_count`

## 5. EvolvableStrategy 抽象接口

### 5.1 接口总览

引擎层只依赖以下动词契约（共 14 个，括号内标注修订来源）：

| Method | Responsibility | 修订 |
|---|---|---|
| `StrategyID()` | 返回策略唯一标识 | — |
| `Segments()` | 返回 `[]SegmentInfo`，含维度边界、量化精度、步长、IsCritical | **问题 B 修订**：从 `CrossoverSegments() [][]int` 改回 |
| `Sample(rng)` | 从合法空间采样 Gene | — |
| `Clamp(g)` | 修复越界与结构约束 | — |
| `Validate(g)` | 检查 Gene 合法性 | — |
| `Crossover(p1, p2, rng)` | 基于 `Segments()` 实施块级正交交叉 | — |
| `Mutate(c, prob, scale, rng)` | 独立 Bernoulli + 高斯扰动；`scale` 是全局 GeneStep 乘数，与 `prob` 独立 | **问题 C 修订**：删除 `eliteStats` 参数 |
| `Fingerprint(c)` | 基于 `SegmentInfo.QuantizationStep` 量化后哈希 | — |
| `Evaluate(ctx, c, plan)` | 在 plan 上评估基因，返回 `*RawEvaluateResult`（**v5.4.1 P09 修订**：原返回 `[]CrucibleResult`。`RawEvaluateResult` 物理上不含 `ScoreTotal`，策略层不可写入聚合值，见 §5.7） | — |
| `ReviewBacktest(ctx, c, spawn, bars, ...)` | 全量历史回顾回测，仅在 Promote 后调用，不影响决策 | **问题 D 修订**：从 `Verify` 重命名 |
| `EncodeResult(c, spawn)` | 编码结果包 | — |
| `DecodeElite(blob)` | 从历史精英结果恢复 Gene | — |
| `MinEvalBars()` | 返回最低有效 bar 数 | — |
| `NewAdapter(plan)` | 为每个 worker 创建独立 Adapter 实例 | **问题 F 修订**：补回缺失的第 14 个动词 |

> **关于 `ResetState()` 的去向（问题 E 修订）：** 原草稿在接口表中列出了 `ResetState()`，但引擎层应通过 Adapter 的 `Reset()` 方法来保证状态隔离，不应要求策略实现方主动调用。正确做法是：策略实现 `NewAdapter(plan) Adapter`，`Adapter.Reset()` 由**引擎在每次 Evaluate 前强制调用**，策略无法绕过。详见 §5.6。

### 5.2 Clamp 语义

Clamp 不仅仅是 clip，上层必须允许其执行：

- 边界裁剪
- 权重和归一化
- 互斥项修复
- 离散档位映射
- 整数窗口取整

### 5.3 Crossover

交叉必须遵守显式 Segment 契约（来自 `Segments()`）。流程：

1. 基于 Segment 整体选取父代来源（对每个 Segment 独立掷 50% 硬币）
2. 执行块级拼接（块内完整继承，不做逐字段混合）
3. 调用 Clamp（顺序：边界裁剪 → 块内约束修复 → 跨段约束修复）
4. 调用 Validate
5. 失败时回退到父代原样拷贝（随机选 p1 或 p2）并记录 `crossover_fallback` 事件

### 5.4 Mutate

基础变异为独立 Bernoulli + 高斯扰动：

```go
// prob 控制变异密度，scale 控制幅度，两者完全独立
for _, seg := range strategy.Segments() {
    for localIdx, geneIdx := range seg.Dimensions {
        if rng.Float64() < prob {
            delta := rng.NormFloat64() * seg.GeneStep[localIdx] * scale
            child[geneIdx] = c[geneIdx] + delta
        }
    }
}
return strategy.Clamp(child)
```

相关变异（对强相关 Segment 使用联合多维高斯采样）为可选增强项，推迟到 v6+。原型阶段不在接口签名中预留 `eliteStats` 参数，避免接口污染。

### 5.5 Evaluate

Evaluate 必须满足：

- 相同 `(Gene, EvaluablePlan)` 必须输出容差范围内一致的结果
- 核心浮点累加不得并发归约；允许串行累加或 Kahan Summation
- 不得读取全局可变状态
- 内部禁止启动 goroutine
- 摩擦参数必须从 `EvaluablePlan.Friction` 读取，禁止硬编码

### 5.6 Adapter 生命周期与状态隔离

> **问题 E 修订：** 原草稿中 `ResetState()` 作为接口动词存在，与 `Adapter.Reset()` 形成双定义。v5.3 修订版统一为：**状态隔离由引擎通过 Adapter.Reset() 保证，不依赖策略实现方主动调用**。

`Adapter` 接口：

```go
type Adapter interface {
    Reset(plan *EvaluablePlan) error
    // v5.4.1 P09 修订：返回 *RawEvaluateResult（不含 ScoreTotal）。
    // 策略层物理上不可写入聚合值；ScoreTotal 由引擎通过 fitness.AggregateScoreTotal 填充。
    Evaluate(gene Gene) (*RawEvaluateResult, error)
    Close() error
}
```

### 5.7 评估与聚合的职责分工（v5.4.1 新增）

GA 流水线的两个阶段严格分开：

| 阶段 | 负责方 | 产出类型 |
|---|---|---|
| 单 gene 多窗口评估 | `Adapter.Evaluate(gene)` (worker) | `*RawEvaluateResult` |
| 级联短路 + 窗口组合 | `EvolvableStrategy.Evaluate(ctx, gene, plan)` | `*RawEvaluateResult` |
| ScoreTotal 聚合 | `fitness.AggregateScoreTotal(...)` (引擎共享函数) | `ScoreTotal` |
| 结果包 evaluation 层组装 | 引擎 `RunEpoch` | `EvaluationLayer` |

**类型级强保证**：`RawEvaluateResult` 不含 `ScoreTotal` 字段，策略永远无法写入。多目标 GA（NSGA-II / Pareto front）等未来演进只需扩展 `EvaluationLayer`，策略层零改动。

引擎调度伪代码：

```go
adapter, _ := strategy.NewAdapter(plan)   // Epoch 启动时，每个 worker 调用一次
defer adapter.Close()

for gene := range workQueue {
    adapter.Reset(plan)                   // 引擎强制，每次 Evaluate 前
    result, _ := adapter.Evaluate(gene)
    results[gene.index] = result
}
```

`Reset` 必须清空的状态：持仓态、指标缓存、交易历史缓存、连续亏损等业务计数器。  
`Reset` 可保留（复用以提升性能）：K 线数据 buffer、DCABaseline 缓存、临时计算 buffer（每次写前自动清空）。

**Reset 不完整会导致评估顺序影响结果，直接破坏容差级确定性承诺。**

## 6. 适应度与评分流水线

### 6.1 适应度目标

GA 代内排序只依赖引擎定义的适应度，不依赖人工审阅指标（DSR、OOS、邻域稳定性等均在代外使用）。

### 6.2 Ghost DCA 双基准

保留双基准：

- 月度定投（GhostDCA-monthly）
- 周度定投（GhostDCA-weekly）

两个基准必须使用与策略相同的摩擦参数（`EvaluablePlan.Friction`）。

### 6.3 ROI 口径

沿用：

- 默认 Modified Dietz
- 注资影响超过 10% 时切换精确 TWR

### 6.4 单窗口评分

计算步骤（对应 v5.2 §3.1.2 双基准公式）：

```
monthly_score = (ROI_s - ROI_m) - λ_diff × max(0, MDD_s - MDD_m)
weekly_score  = (ROI_s - ROI_wk) - λ_diff × max(0, MDD_s - MDD_wk)
base_score    = min(monthly_score, weekly_score)
                - λ_abs × max(0, MDD_s - DD_base)²
turnover_penalty = λ_turn × max(0, T_annual - T_extreme)³
slice_score   = base_score - turnover_penalty
```

各步骤独立输出，方便诊断字段分层记录（`monthly_score`、`weekly_score`、`base_score`、`turnover_penalty` 均写入报告）。详细参数默认值见 v5.2 §3.1 和 §3.1.5。

### 6.5 Fatal 规则与级联短路

> **问题 A 修订：** 原草稿 §6.5 写"仍继续评估其它窗口以保留诊断信息"，这取消了级联短路机制。v5.3 修订版恢复级联短路，并通过抽样审计获取诊断信息，两者不冲突。

**默认行为：级联短路（从短到长 6m → 2y → 5y → 10y）。**

一旦某个窗口触发 Fatal（`MDD_s >= FatalMDD`）：

- 该窗口 `SliceScore.Fatal = true`
- **立即终止**，跳过后续更长窗口（节省约 70% 计算量，早期代 fatal 率通常 40-60%）
- `ScoreTotal` 直接记为 Fatal（Sum-type 传染规则）

**诊断信息的获取途径：** 对每代 fatal 个体随机抽取 `fatal_audit_sample_rate`（默认 5%）进行**完整四窗口评估**，结果写入 audit 表，**不参与本代 GA 排序**。这是"短路不丢诊断"的平衡方案，计算代价极低。

Fatal 时记录诊断元数据（无额外计算成本）：

```go
type CrucibleResult struct {
    Window        string    // 枚举："6m" / "2y" / "5y" / "10y"
    Score         SliceScore
    FatalReason   *string   // 仅 Score.Fatal=true 且 SkippedBy=nil 时有意义
    FatalAtBar    *int64    // fatal 发生时的 bar 时间戳（UTC 毫秒）
    FatalMDDValue *float64  // 触发 fatal 时的具体 MDD 值
    BarsEvaluated int       // 被短路跳过的窗口此值为 0
    SkippedBy     *string   // 非 nil 时表示被级联短路跳过，如 "cascaded_from_6m"
                            // 此时 Score.Fatal=false, Score.Value=nil, BarsEvaluated=0
    Components    *CrucibleScoreComponents // skipped_by 非 nil 时此字段为 nil
}
```

> **Go 实现注意（Fatal 排序 Panic 风险）：** `Score.Fatal=true` 时 `Score.Value` 为 nil。被短路跳过的窗口 `Score.Value` 也为 nil。排序时直接解引用 nil `*float64` 会 Panic。引擎**必须**通过封装的 `CompareFitness` 函数排序（见 §2.6），禁止向 `Value` 写入哨兵数值。对应测试：`TestCompareFitnessNilSafe`（§10.1 第 11 条）。

### 6.6 一致性惩罚

使用窗口间 dispersion 惩罚。

**原型阶段默认使用原始 `SliceScore` 标准差**（实现更简单）：

```
ConsistencyPenalty = λ_cons × σ({SliceScore_w.Value | w ∈ Windows, Score.Fatal=false, Score.SkippedBy=nil})
```

如任一窗口 Fatal，跳过一致性惩罚（Fatal Sum-type 传染优先）。被短路跳过的窗口（`SkippedBy` 非 nil）也不参与 σ 计算。

**原型阶段 λ_cons 默认值：0.3**（来源：工程经验，与 v4 保持向后兼容，`fitness_version = "v1-raw-std"`；待实证后调整）。

**升级路径：** v5.2 已设计 z-score 标准化方案（消除量纲偏置，λ_cons 对应调整为 0.02）。从原始标准差迁移到 z-score 时，通过 `fitness_version` 版本号标记，避免跨版本分数直接比较。

> **结论：** 原型阶段用 `fitness_version = "v1-raw-std"`，`λ_cons = 0.3`；切换 z-score 后用 `fitness_version = "v2-zscore"`，`λ_cons = 0.02`。

### 6.7 OOS / ReviewBacktest / Promote 角色边界

> **问题 D 修订：** 将 `Verify` 统一改为 `ReviewBacktest`，区分角色。

| 角色 | 数据范围 | 触发时机 | 影响 GA 排序 | 影响 Promote 决策 |
|---|---|---|---|---|
| OOS 回测 | 固定 Anchored Holdout 段 | Epoch 结束后 | 否 | 是（参考） |
| ReviewBacktest | IS + OOS + Sacred 全量 | Promote 后用户触发 | 否 | 否（仅展示） |
| Promote | — | 人工审批 | 否 | 是（最终决定） |

OOSWindow 在 Epoch 创建时冻结，
后续 Epoch 不得滚动进入未来 IS，
以防人工 Promote 构成 meta-optimization leakage。

### 6.8 DSR 定位

DSR 仅为人工 Promote 辅助指标，不得参与 GA 代内适应度排序。

### 6.9 脆性与敏感度

原型阶段保留两个扩展指标：

- SpawnPoint 扰动敏感度
- 参数邻域脆性检测（高斯邻域 N=30 采样 + IsCritical Segment 的 OAT 扰动）

可先不阻塞主流程，只需在报告中预留字段；字段内容二期填充。

## 7. 并发、确定性与缓存

### 7.1 Worker Pool

默认：

```text
Workers = min(NumCPU, PopulationSize)
```

每个 worker 独占一份 Adapter（通过 `NewAdapter(plan)` 创建），不共享可变状态。

### 7.2 RNG 边界

- 主线程负责 Tournament / Crossover / Mutation 的主 RNG（从 EpochSeed 派生）
- **worker 内不得消费主 RNG**
- 如策略 Evaluate 内部需要随机过程，必须从 `EpochSeed` 派生独立的确定性 RNG

### 7.3 顺序无关性

必须保证：

- `results[i]` 只对应 `population[i]`，不与 worker 完成顺序相关
- 改变评估顺序不应改变结果（`TestEvaluateOrderInvariance` 验证此约束）

### 7.4 浮点确定性

必须避免并发归约导致的浮点不稳定。  
原型阶段承诺"容差级重放"优先于"位级绝对一致"：

- ScoreTotal 差异 ≤ 1e-6
- Chromosome 字段差异 ≤ 1e-9（绝对）

跨硬件 / 跨编译产物的可复现性不做承诺，`hardware_signature` 和 `go_version` 写入 challenger 元数据备查。

### 7.5 Genome 指纹缓存

Fingerprint 作为 Epoch 内重复评估缓存键。  
量化精度来自 `SegmentInfo.QuantizationStep`，引擎从 `Segments()` 读取，不依赖全局统一精度。

原型阶段：

- 先支持 exact 量化哈希
- DSR 近亲判定（fingerprint_distance 用于独立试验 N 的计数）可后续引入单独距离度量

### 7.6 `bars_hash` 序列化边界（v5.4.1 P12 固化）

`bars_hash` 的序列化范围**仅包含** `Bar` 结构的价格数据字段：

```
OpenTime, Open, High, Low, Close, Volume（完整 OHLCV）
```

`Bar.IsGap` / `Bar.GapType` 是元数据字段，**不参与 `bars_hash`**。

**理由**：缺口检测算法升级（改变 `IsGap` 标记）与底层价格数据无关，不应破坏基于价格数据的可复现性承诺。数据与元数据分离，哈希契约稳定。

该边界约定写入 `/internal/quant/canonical_json.go` 文件顶部注释，作为项目级冻结契约。对应测试：`TestBarsHashExcludesMetadata`。

v5.3 最小结果包：

├── core
├── evaluation
├── verification

## 10. 测试与质量保证

### 10.1 必须优先落地的测试

原型阶段必须先落以下测试：

1. `TestEvaluateDeterministic`（相同输入两次结果在容差内一致）
2. `TestEvaluateOrderInvariance`（100 个 gene 正序 vs 倒序结果在容差内一致）
3. `TestAdapterResetIsolation`（Reset 不完整时结果应失败，验证隔离有效）
4. `TestClampValidateContract`（Clamp 后调用 Validate 必须通过）
5. `TestSegmentsCoverage`（Segments() 覆盖所有维度，无遗漏无重复，三数组等长）
6. `TestCrossoverBlockFidelity`（child 的每个 Segment 完整等于 p1 或 p2）
7. `TestReplayWithinTolerance`（同 EpochSeed、同数据版本结果在容差内）
8. `TestGapHandlingNoFakeTrades`（IsGap=true 时无虚假成交，indicator 跳过该 bar）
9. `TestMutationScaleLinearity`（scale=2.0 时平均变异量约为 scale=1.0 的 2 倍）
10. `TestCascadeShortCircuit`（6m Fatal 时后续窗口 SkippedBy 字段非空，ScoreTotal.Fatal=true）
11. `TestCompareFitnessNilSafe`（**v5.3.3 新增**，验证 CompareFitness 函数在所有 Fatal/Normal 组合——含双 Fatal——下均不 Panic，且排序稳定）

> **测试 11 新增背景：** `SliceScore.Value` 为 `*float64`，`Fatal=true` 或被短路跳过时值为 nil。Go `sort.Slice` 对 nil 指针直接解引用会 Panic。此测试强制验证引擎已通过封装的 `CompareFitness` 函数规避此风险。

### 10.2 回放测试

至少验证：

- 同 EpochSeed、同数据版本、同配置下结果在容差内一致
- 结果包可重新驱动评估链路

### 10.3 API 契约测试

必须验证：

- 任务创建参数校验（含 `spawn_mode` 枚举约束、`test_mode` boolean 类型、`oos_days` 最小值 > 0）
- 状态流转正确性
- Promote 仅允许 challenger 状态的记录执行
- `test_mode=true` 时结果包中 `friction_disabled=true` 且不可 Promote

## 11. 参数治理与版本管理

### 11.1 参数分类

建议沿用 v5.2 §3.7 的思路，将参数标注为：

| 类别 | 含义 | 可修改性 |
|---|---|---|
| 物理常数 | 数学/算法定义（如 Euler-Mascheroni γ=0.5772） | 不可改 |
| 工程约束 | 确定性/隔离等工程硬约束（如禁止并发归约） | 不可改 |
| 设计选择 | 价值判断（如 FatalMDD=0.70、DD_base=0.40） | 中 |
| 工程经验 | 待实证标定（如 λ_turn=0.1、邻域采样 N=30） | 高 |

### 11.2 三类版本号

v5.3 起显式维护：

| 版本号 | 管理对象 | 变更时机 |
|---|---|---|
| `schema_version` | 结果包 JSON 结构 | 新增 / 删除 / 重命名字段 |
| `fitness_version` | 适应度公式与参数 | 公式变更、λ 调整、z-score 切换 |
| `fingerprint_version` | 指纹量化逻辑 | QuantizationStep 调整、哈希算法变更 |

跨版本比较时：`schema_version` 不同允许字段兼容读取；`fitness_version` 不同的 challenger 分数**不应直接比较**（公式语义已变），前端应标注版本标签。

### 11.3 可调参数与不可调参数

不可调（工程约束级）：

- 哈希算法定义
- 接口必需动词（14 个，含 `NewAdapter`）
- 禁止并发归约
- Evaluate 内部禁止 goroutine

可调（设计选择 / 工程经验级）：

- `FatalMDD`、`lambda_turn`、`T_extreme`
- 窗口权重（原型阶段保持默认）
- 救援阈值与重启比例
- `fatal_audit_sample_rate`

### 11.5 配置快照语义（v5.4.1 P11 新增）

`GAConfigSnapshot`（写入 `core.ga_config`）的字段语义为**生效值**，不是请求镜像：

- `test_mode=true` 时，`taker_fee_bps` 与 `slippage_bps` 写入 **0**（引擎写快照前覆写）。
- `test_mode=false` 时，`taker_fee_bps` 与 `slippage_bps` 为用户请求值。

**不变式**（可由 `GAConfigSnapshot.Validate()` 校验）：

```
test_mode == true  →  taker_fee_bps == 0 && slippage_bps == 0
evaluation.friction_actual.taker_fee_bps == ga_config.taker_fee_bps  (数值一致)
```

**用户原始请求意图**（如 `test_mode=true` 但请求 `taker_fee_bps=10`）由 `EvolutionTask` 表的 `requested_taker_fee_bps` / `requested_slippage_bps` 字段单独记录，不进入结果包。

这一设计消除了旧版 `{test_mode:true, taker_fee_bps:10}` 并存的逻辑自相矛盾，使结果包 JSON 值始终自洽。

### 11.4 版本迁移策略

原型阶段约定：

- 新系统读旧结果包时允许缺失字段填默认值
- 旧系统读新结果包时忽略未知字段
- `fitness_version` 不同的结果包不进行跨版本分数排名
- 大版本升级时单独提供迁移脚本
