# 进化系统 v5.3.3 Go-only JSON Schema

**用途：** 本文档根据《进化系统_程序框架规划_v5.3（修订版）》定义原理验证 / 原型迭代期在 **Go-only 服务器端方案** 下的外部数据契约，重点覆盖 HTTP API 请求响应结构、数据库 JSON 持久化对象，以及 `ChallengerResultPackage` 结果包结构。

**范围：** 本文档只约束会穿越系统外部边界的数据对象，包括 API 请求响应、结果包、数据库 JSON blob、回放与版本兼容所需的元数据；Go 进程内部的运行时对象与服务内接口，如 `Adapter` 实例、worker 上下文、纯内存 `AggregateCache` 和内部领域对象传递，不属于本文档直接约束范围。

**定位原则：** 在 Go-only server prototype 下，领域模型优先以 Go struct 和 interface 实现；仅当对象需要穿越 HTTP API、数据库 JSON、结果包、回放 / 审计 / 兼容读取等边界时，才要求定义稳定的 JSON schema。原型阶段 OOS 统一采用 **Anchored OOS Holdout** 语义（固定往前推若干天），不再保留 rolling OOS 的字段或命名。

> **v5.3.3 相对 v5.3.1 的修订摘要（见附录 A 完整列表）：**
> - `CreateEvolutionTaskRequest` / `GAConfigSnapshot`：`oos_config` 对象及其 `rolling_oos_days` / `enabled` 字段，统一替换为顶层 `oos_days` 整型；`spawn_mode` 冻结为枚举；`test_mode` 收紧为 boolean；`pair` 补入 `GAConfigSnapshot`。
> - `ChampionGenePayload.encoding` 收紧为原型阶段仅允许 `"json"`。
> - `ReproducibilityMetadata` 补充 `plan_hash` / `bars_hash` 的算法约定。
> - `PromoteLayer.decision_status` 删除 `retired`（该状态属于 Champion 生命周期，不属于 Challenger 结果包）。
> - `SliceScore` / `CrucibleResult` 补充被短路跳过窗口的校验语义。
> - 新增 §2.4「Fatal 排序语义」，明确 Fatal 个体在 Go 中的比较方案。

---

## 0. 设计原则

1. **边界优先**：JSON schema 用于约束外部可见数据，而不是替代 Go 内部类型系统。
2. **最小持久化**：不为纯运行时对象强行设计 JSON 结构，优先减少无意义序列化。
3. **版本显式化**：结果包必须显式记录 `schema_version`、`fitness_version`、`fingerprint_version`，且 `fitness_version` 不同的结果不直接比较。
4. **可重放优先**：所有进入结果包和数据库的关键对象都要服务于 replay、诊断和兼容读取。
5. **Go-only 简化**：原型阶段不再以 Python / Go 共享模型为主目标，因此 `EvaluablePlan`、`Bar`、`CrucibleWindow` 等内部对象默认优先保留为 Go struct，除非确有持久化需求。

## 2. 命名与通用约定

### 2.1 命名风格

- JSON 字段统一使用 `snake_case`。
- 枚举值统一使用 `snake_case`。
- 时间戳字段统一采用 UTC 毫秒整型，并以 `_ts` 结尾。
- 版本字段统一为字符串。

### 2.2 统一枚举建议

为减少自由文本漂移，以下枚举集合在原型阶段冻结。

```json
{
  "decision_status": ["pending", "promoted", "rejected"],
  "run_status": ["queued", "running", "succeeded", "failed", "cancelled"],
  "verification_status": ["not_run", "ok", "failed", "insufficient_data"],
  "decision_color": ["green", "yellow", "red", "gray"],
  "skipped_by": ["cascaded_from_6m", "cascaded_from_2y", "cascaded_from_5y"],
  "spawn_mode": ["inherit", "random_once", "manual"]
}
```

> **关于 `decision_status` 移除 `retired`：** `retired` 属于 Champion 实体的生命周期状态，不属于 Challenger 结果包语义。Challenger 的决策状态只有三种：`pending`（待审批）、`approved`（晋升为 Champion）、`rejected`（不晋升）。Champion 的退役在独立的 champion_history 表中管理。

### 2.3 版本常量建议

```json
{
  "schema_version": "v5.3.3",
  "fitness_version": "v1-raw-std",
  "fingerprint_version": "fp-v1"
}
```

### 2.4 Fatal 排序语义（v5.3.3 新增）

`SliceScore` 和 `ScoreTotal` 采用 `{fatal: bool, value: number|null}` 结构。当 `fatal=true` 时 `value` 为 `null`，这对 JSON 序列化是合法的，但在 Go 的 `sort.Slice` 中直接解引用 `*float64` nil 指针会引发 Panic。

**引擎层必须封装专用比较函数，而不是直接对 `value` 字段做大小比较：**

```
比较规则（三层优先级）：
  Normal > Fatal              // Fatal 个体永远排在最后
  Fatal vs Fatal → 平局       // 用 Fingerprint 字典序做稳定排序
  Normal vs Normal → 按 value 降序
```

**禁止** 引擎通过向 `value` 写入 `-99999` 或 `-1e18` 等哨兵数值来规避 nil 判断。所有比较必须通过封装的 `CompareFitness(a, b)` 函数进行，该函数必须在调用 `*value` 之前显式检查 `fatal` 标志。

此约束不体现在 JSON schema 的值域限制中，而是对引擎实现的行为约束。对应测试：`TestCompareFitnessNilSafe`。

## 4. API Schema

### 4.1 CreateEvolutionTaskRequest

对应 `POST /api/v1/evolution/tasks` 的最小请求体。

> **v5.3.3 修订：**
> - 删除 `oos_config` 嵌套对象（移除了 `enabled *bool` 的三态逻辑和非法 JSON tag 问题）。
> - 新增顶层 `oos_days`：`null` 或缺省表示不启用 OOS；`> 0` 整数表示从最新 bar 往前推固定天数作为 Anchored Holdout 窗口。
> - `spawn_mode` 冻结为枚举 `["inherit", "random_once", "manual"]`。
> - `test_mode` 收紧为 `boolean`（`true` = 烟雾测试模式，强制 Pop=10/Gen=3/摩擦禁用）。

```json
  "required": [

    "max_generations",
    "elite_ratio",
    "fatal_mdd",
    "taker_fee_bps",
    "slippage_bps",
    "spawn_mode",
    "test_mode"
  ],
  "properties": {
    "strategy_id": { "type": "string", "minLength": 1 },
    "pair": { "type": "string", "minLength": 1 },
    "pop_size": { "type": "integer", "minimum": 1 },
    "max_generations": { "type": "integer", "minimum": 1 },
    "elite_ratio": { "type": "number", "minimum": 0, "maximum": 1 },
    "fatal_mdd": { "type": "number", "minimum": 0, "maximum": 1 },
    "taker_fee_bps": { "type": "number", "minimum": 0 },
    "slippage_bps": { "type": "number", "minimum": 0 },
    "spawn_mode": {
      "type": "string",
      "enum": ["inherit", "random_once", "manual"],
      "description": "inherit = 继承当前 champion 的 SpawnPoint；random_once = 任务排队时随机生成一次并冻结；manual = 由请求体中的 spawn_point 字段指定"

    },
    "oos_days": {
      "type": ["integer", "null"],
      "minimum": 1,
      "description": "Anchored OOS Holdout 天数。null 或缺省 = 不启用 OOS；> 0 = 从最新 bar 往前推指定天数作为固定 Holdout 窗口，Epoch 创建时冻结，后续不滚动"
    },
    "fatal_audit_sample_rate": {
      "type": ["number", "null"],
      "minimum": 0,
      "maximum": 1,
      "description": "每代 fatal 个体的完整四窗口评估抽样比例，默认 0.05；仅用于诊断，不参与 GA 排序"
    },
    "spawn_point": {
      "type": ["object", "null"],
      "description": "仅当 spawn_mode = manual 时使用",
      "additionalProperties": true
    }
  },
  "additionalProperties": false
}
```

## 11. 实施建议

### 11.1 第一阶段必须冻结的契约

建议先冻结以下对象的字段与枚举，因为它们直接影响 API 契约测试、Replay、结果包兼容和 Promote 流程：

- `PromoteLayer`
- `ReproducibilityMetadata`

### 11.2 第一阶段允许保留宽松对象的部分

原型阶段以下字段可先保持宽松 `object`，等服务主链路跑通后再细化：

- `alpha_breakdown`
- `dsr_summary`
- `stress_summary`
- `turnover_metrics`
- `clamp_modifications`
- `diversity_rescue_log`
- `mutation_ramp_log`

### 11.3 与测试计划的对应关系

该 schema 文档应至少服务于以下测试目标：

- `TestCascadeShortCircuit`：要求后续窗口 `skipped_by` 非 null，且 `score_total.fatal=true`，`score.value=null`。
- `TestCompareFitnessNilSafe`：要求引擎的 `CompareFitness` 函数在任意 Fatal/Normal 组合下不 Panic（含双 Fatal 情形）。
- `TestReplayWithinTolerance`：要求结果包中包含足够的重放元数据（`plan_hash` / `bars_hash` / `epoch_seed`）。
- API 契约测试：验证任务创建参数校验（`spawn_mode` 枚举、`test_mode` boolean 类型、`oos_days` 取值）、状态流转和 Promote 条件。
- `TestEvaluateDeterministic` 与 `TestEvaluateOrderInvariance`：虽然核心发生在 Go 内部，但相关输出结构必须稳定可比较。
